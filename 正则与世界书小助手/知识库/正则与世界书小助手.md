# 第一部分：核心机制 — 格式标识符

## 1. 什么是格式标识符

前端 SillyTavern 通过正则表达式匹配 AI 输出中的特定标记，将其替换为渲染后的 HTML。世界书条目的职责是**强制 AI 输出带有这些标记的格式化文本**。

```
世界书条目 content → 约束 AI 输出格式 → AI 输出带标识符的文本 → 前端正则匹配 → HTML 渲染
```

因此世界书的内容写作本身就是"正则的第一环"——它定义了正则要捕获的目标格式。

## 2. 格式标识符的三种实战形态

### 形态一：Start/End 包裹器

**语法**：`标识符_start` / `标识符_end`

**用途**：包裹整块需要渲染的内容区域。

**示例结构**：
```
xxx_start
[具体内容区域]
xxx_end
```

**关键**：必须声明"确保只有一个 xxx_start 和 xxx_end"、"以 xxx_end 收尾"。

### 形态二：XML 风格标签

**语法**：`<标签名>` / `</标签名>`

**用途**：包裹特定功能模块，或标记需要渲染/过滤的内容块。

**可内嵌于 Start/End 包裹器内部**，形成嵌套结构。

### 形态三：管道分隔 + 特殊符号标记

**语法**：
- 单行管道：`{标记|字段1|字段2|字段3}`
- 多行键值：`字段名:值`
- 双短横分隔：`字段A--字段B--字段C`
- 方括号格式：`[类型-内容]`

**用途**：结构化数据输出，便于前端按字段解析。

## 3. 占位符约定

| 符号 | 用途 | 输出规则 |
|------|------|----------|
| `${变量}` | 动态填充的占位符 | **禁止输出 `${}` 符号本身** |
| `{{变量}}` | 角色名/用户名等系统变量 | 替换为实际值后输出 |
| `{{//文本}}` | 给人类读者的注释 | **禁止输出**到 AI 回复中 |
| `<随机标识>` | 动态生成的值 | 替换为实际随机值后输出 |

# 第二部分：Content 内容布局模式

## 模式一：指令头 + 格式体 + 字段说明 + 示例

**结构顺序**：
```
1. 强制性指令头（"注意！！你必须..."）
2. 格式体（实际输出模板）
3. 占位符声明（"以上所有 ${} 都是占位符，禁止输出"）
4. 字段逐一定义（"  - 字段名：具体要求"）
5. 格式示例（"以下仅为格式示例，输出不可照搬"）
```

**适用场景**：状态栏、信息面板、任何需要固定格式输出的条目。

## 模式二：标识符声明 + 子格式块 + 特殊消息类型 + 闭合规则

**结构顺序**：
```
1. 外层标识符声明（xxx_start / xxx_end）
2. 子容器格式（<子容器></子容器>）
3. 数据行格式（字段分隔符约定）
4. 特殊消息类型列表（[type-内容] 格式全表）
5. 格式闭合规则（"确保标签闭合"、"请勿生成多个 xxx_start"）
```

**适用场景**：聊天系统、论坛、任何多子模块的格式化输出。

## 模式三：XML 包裹 + 内部混合子格式

**结构顺序**：
```
<外层标签>
# 可选 Markdown 标题
规则用 - 列表
## 子规则用 ## 标题
YAML 缩进 / JSON 数据块
</外层标签>
```

**适用场景**：行为规范、NSFW 描写指导、世界观设定。

## 模式四：纯 YAML 嵌套

**结构顺序**：
```
顶层键:
  二级键:
    三级键:
      - 列表项1
      - 列表项2
  另一个二级键:
    描述文本
```

**核心规则**：
- 冒号后跟一个空格
- 子级缩进两个空格
- 列表用 `- ` 开头
- 不混合使用 Tab 和空格

**适用场景**：人物设定、NPC 档案、世界观背景。

## 模式五：$ 命令触发

**结构顺序**：
```
当{{user}}输入："$命令名"时，[动作描述]。
[输出格式要求]。
[后续动作]。末尾必须生成格式正确的完整的状态栏。
```

**关键要素**：
- 第一句必须写明触发条件和动作
- 输出格式要求要具体（HTML/纯文本/JSON 等）
- 链接触发：末尾明确要求"必须生成状态栏"以防止遗漏

**适用场景**：需要用户主动指令触发的系统功能。

# 第三部分：指令语言编写规范

## 1. 强制性层级

| 层级 | 典型句式 | 使用位置 |
|------|---------|---------|
| 最强 | "注意！！你必须在每次回复的正文尾部..." | 格式条目开头 |
| 强 | "必须"、"禁止"、"不得"、"严格" | 核心规则 |
| 中 | "确保"、"请勿"、"仅允许" | 格式约束 |
| 奖惩 | "做得好我会给你XXX奖励"、"违反则XXX" | 格式条目末尾 |

## 2. 正向表述原则

```
正向（正确）："一次生成10-15条内容，每条包括id"
负向（错误）："不要只生成几条"

正向（正确）："内容如果需要换行使用<br>"
负向（错误）："不要用太多换行"

正向（正确）："除非遇到重大事故，否则尽可能完整播完整场"
负向（错误）："不要中途停止直播"
```

## 3. 数量/范围限定原则

凡是可量化的要求，必须给出具体数字：
- "一次生成10-15条" 不是 "若干条"
- "控制在15字以内" 不是 "简短一些"
- "每条动态2-4条评论" 不是 "适量评论"
- "字数不少于3000字" 不是 "写长一点"

## 4. 示例防护原则

任何格式示例前必须加免责声明：
- "以下仅为状态栏示例，不得照搬照抄"
- "仅为格式示例，输出不可照搬"
- "参考示例："（适合可参考但不强制的内容）

## 5. 链接触发原则

涉及多个格式块的条目，末尾必须链接触发：
- "末尾必须生成格式正确的完整的状态栏"
- "首尾必须用```html和```包裹"

# 第四部分：参数使用规律

## 1. constant + selective 组合模式

实战中观察到的四种组合及适用场景：

| constant | selective | 含义 | 典型场景 |
|----------|-----------|------|---------|
| true | true | 始终激活 + keys 辅助 | 人物设定、行为规范、语言规范 |
| true | false | 始终激活，不需要 keys | 状态栏、顶部信息栏 |
| false | true | keys + secondary_keys 同时命中触发 | 特殊场景切换 |
| false | false | keys 精确匹配触发 | 地点类、NPC类、系统指令类 |

## 2. position 选择规则

| position | 含义 | 适用条目 |
|----------|------|---------|
| before_char | 注入在角色描述之前 | 世界观背景、地理格局、角色基础信息列表 |
| after_char | 注入在角色描述之后 | 行为规则、状态栏、NSFW、格式规范 |

**判断标准**：条目内容是"角色的基础设定信息"还是"对角色行为的约束指令"。前者用 before_char，后者用 after_char。

## 3. insertion_order 排序原则

**核心原则**：数值越大，注入越靠后，越靠近 AI 的"眼前"，对输出的直接约束力越强。

- **格式约束类** → order 最高（直接决定输出形态，约占总条目数的末尾 20%）
- **行为规则类** → order 中等偏高
- **状态面板类** → order 中高
- **人物设定类** → order 中等
- **世界观背景类** → order 较低（先注入成为背景知识）
- **隐藏/剧透类** → order 最低（最先注入，沉在上下文底部）

## 4. enabled 决策

- `enabled: true`（必开）：正常游戏流程需要的条目
- `enabled: false`（选开）：剧透内容、隐藏身世、特殊剧本

# 第六部分：质量检查清单

生成完毕后，自动核验：

**格式闭合**：
- [ ] Start/End 标识符是否成对（xxx_start ↔ xxx_end）
- [ ] XML 标签是否正确闭合
- [ ] 是否声明"确保只有一个 xxx_start"

**占位符**：
- [ ] `${}` 是否标注为"禁止输出"
- [ ] `{{}}` 是否区分了系统变量和人类注释

**指令质量**：
- [ ] 格式条目是否有奖惩句式
- [ ] 数量要求是否具体数字
- [ ] 规则是否用正向表述
- [ ] 格式示例前是否声明"不得照搬"

**参数**：
- [ ] position 是否正确
- [ ] insertion_order 是否与条目功能匹配
- [ ] 选开条目是否 enabled: false
- [ ] constant/selective 组合是否匹配触发方式

**完整性**：
- [ ] 多模块条目是否有链接触发（"末尾必须生成状态栏"）
- [ ] 特殊格式是否附带完整的使用范围说明

# 第七部分：常见错误与避免

| 错误 | 后果 | 正确做法 |
|------|------|----------|
| 标识符未闭合 | 前端无法匹配 | 每对 xxx_start/xxx_end 必须成对 |
| `${}` 未声明禁止输出 | AI 原样输出 `${时间}` | 单独一行声明 |
| 示例未标注"不得照搬" | AI 每次复制相同内容 | 加免责声明 |
| 模糊表述 | 输出质量不稳定 | 用具体数字和范围 |
| 格式条目 order 过低 | 格式约束被背景覆盖 | 格式类 order 最高 |
| 未加奖惩句式 | 格式遵守率低 | 格式类条目必加 |
| 未链接触发 | 多个格式块遗漏 | 末尾明确"必须生成状态栏" |

# 第八部分：工作流程

## 步骤1：识别需求

判断用户需要的产出类型：
- 需要**固定格式输出**？→ 模式一/二（指令头+格式体）
- 需要**触发后执行动作**？→ 模式五（$ 命令）
- 需要**约束行为**？→ 模式三（XML 包裹）
- 需要**存储结构化数据**？→ 模式四（YAML 嵌套）

## 步骤2：确定参数

根据触发方式确定 constant/selective 组合，根据内容性质确定 position，根据约束力需求确定 insertion_order。

## 步骤3：编写 content

按照第二部分对应模式的结构顺序编写，遵循第三部分的指令语言规范。

## 步骤4：质量检查

按第六部分清单逐项核验。
